# Требования к внешним интеграциям - Сервисы Умного дома

## 1. Требования к безопасности

**1.1 Шифрование данных**
- Все соединения между PropDevelopment и платформой партнера должны использовать TLS 1.3.
- Передача данных в открытом виде запрещена.
- Данные событий доступа (журналы открытий домофона/шлагбаума) должны храниться в зашифрованном виде.

**1.2 Защита биометрических данных**
- Биометрические шаблоны (слепки лиц) и данные номеров автомобилей обрабатываются только платформой партнера.
- PropDevelopment передаёт партнеру идентификаторы пользователей и разрешения - не сырые биометрические данные.
- Список лиц и номеров автомобилей, имеющих доступ, формируется в PropDevelopment и передаётся партнеру только по защищённому каналу.

**1.3 Изоляция данных арендаторов**
- Данные жильцов каждого жилого комплекса должны быть логически изолированы на стороне партнера.
- API-контракты не должны допускать получение или изменение данных жильцов другого ЖК через ошибки авторизации.
- Каждый запрос к API партнера должен содержать идентификатор ЖК, к которому относится действие.

**1.4 Минимизация данных**
- API-контракты передают партнеру только данные, необходимые для выполнения конкретной функции.
- Персональные данные (ФИО, паспортные данные, контакты) не передаются в запросах к API партнера.
- Партнёр получает только идентификатор пользователя PropDevelopment и разрешение на доступ - не профиль пользователя.

**1.5 Журналирование**
- Каждое событие открытия домофона или шлагбаума (ручное и автоматическое) записывается в журнал с временной меткой, идентификатором пользователя и идентификатором устройства.
- Запросы к API партнера и ответы на них логируются на стороне PropDevelopment.
- Журналы хранятся не менее 90 дней и доступны для аудита безопасности.

**1.6 Защита от атак**
- smart-home-service реализует rate limiting: не более 10 запросов в минуту с одного пользователя на операции открытия устройств.
- Webhook-запросы от партнера принимаются только с заранее согласованных IP-адресов.
- Повторные атаки (replay attacks) блокируются через проверку timestamp в подписи запроса (окно валидности - 5 минут).

**1.7 Соответствие законодательству**
- Биометрические персональные данные относятся к специальной категории по ФЗ-152. Требуется явное письменное согласие собственника на обработку биометрии.
- Партнер должен иметь статус оператора персональных данных согласно ФЗ-152 и обеспечивать хранение биометрии на серверах в РФ.
- Перечень передаваемых партнеру данных фиксируется в договоре и согласуется со специалистом по ИБ PropDevelopment.

## 2. Протоколы аутентификации и авторизации

**2.1 Аутентификация собственников (пользователь -> PropDevelopment)**
- Протокол: OpenID Connect (OIDC) поверх OAuth 2.0.
- Реализация: существующий Keycloak (auth-service-2) выдаёт JWT access token после успешной аутентификации в мобильном приложении.
- JWT-токен содержит claims: идентификатор пользователя, роль, идентификатор ЖК.
- Срок жизни access token - 15 минут; refresh token - 7 дней.

**2.2 Авторизация запросов к smart-home-service (PropDevelopment)**
- smart-home-service проверяет подпись и срок действия JWT-токена через публичный ключ Keycloak (без сетевого запроса).
- Доступ к управлению домофоном или шлагбаумом разрешён только собственникам, привязанным к конкретному ЖК.
- Права доступа:
  - `smarthome:intercom:open` - ручное открытие домофона
  - `smarthome:barrier:open` - ручное открытие шлагбаума
  - `smarthome:access:manage` - управление списком разрешённых лиц и автомобилей (только для администратора ЖК)

**2.3 Аутентификация к API партнера (сервис-к-сервису)**
- Протокол: OAuth 2.0 Client Credentials Flow.
- PropDevelopment зарегистрирован как клиент на стороне партнера с ограниченными scopes.
- Клиентские учётные данные (client_id, client_secret) хранятся в защищённом хранилище секретов (HashiCorp Vault или Kubernetes Secret).
- Полученный access token кешируется в памяти smart-home-service до истечения срока действия.
- Токен доступа к API партнера не передаётся клиентским приложениям и не логируется.

**2.4 Верификация Webhook-событий от партнера**
- Партнер подписывает каждый входящий запрос с помощью HMAC-SHA256 и секретного ключа, согласованного при подключении.
- PropDevelopment проверяет подпись перед обработкой события.
- Запросы без корректной подписи отклоняются с кодом 401.

## 3. Организация взаимодействия

**3.1 Архитектурный паттерн**
- Вводится новый компонент `smart-home-service` (адаптер) в группе сервисов ЖКУ PropDevelopment.
- Сервис инкапсулирует всё взаимодействие с внешней платформой партнера.
- Мобильное приложение не обращается к API партнера напрямую - только через `smart-home-service`.
- Остальные компоненты PropDevelopment (tenant-core-app, CRM) не имеют прямого доступа к API партнера.

**3.2 Синхронное взаимодействие (команды управления)**
- Сценарии: ручное открытие домофона/шлагбаума собственником, управление списком доступа.
- Поток: мобильное приложение -> smart-home-service -> API партнера -> устройство.
- Протокол: REST/HTTPS, формат данных: JSON.
- Таймаут ожидания ответа от партнера: 5 секунд. При превышении - возврат ошибки пользователю.

**3.3 Асинхронное взаимодействие (события доступа)**
- Платформа партнера уведомляет PropDevelopment о событиях (автоматическое открытие домофона по лицу, открытие шлагбаума по номеру) через Webhook (HTTP POST).
- Endpoint для приёма событий доступен только через PropDevelopment API Gateway.
- После проверки подписи события записываются в журнал доступа и могут отображаться в мобильном приложении собственника.

**3.4 Версионирование и управление изменениями API**
- API-контракт с партнером фиксируется в конкретной версии (v1) и описывается в OpenAPI-спецификации.
- Изменения в контракте согласуются с командой безопасности PropDevelopment до внедрения.
- При необходимости обновления обеспечивается период параллельной поддержки старой и новой версии.

**3.5 Отказоустойчивость**
- Сбои платформы партнера не влияют на основные функции мобильного приложения (ЖКУ, информация о ремонте, оплата).
- `smart-home-service` реализует circuit breaker: при трёх последовательных ошибках API партнера переходит в режим ограниченной функциональности с понятным сообщением пользователю.
- Метрики доступности и задержки API партнера собираются в систему мониторинга PropDevelopment.
